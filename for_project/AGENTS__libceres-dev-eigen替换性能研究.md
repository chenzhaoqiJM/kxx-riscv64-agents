- 不要在本机执行编译、测试、基准测试、`cmake`、`make`、`ninja`、`dpkg-buildpackage`、`apt`、`pkg-config` 等会依赖本机环境的命令。
- 不要参考本机头文件、库文件、包版本、编译器输出或 `/usr/include`、`/usr/lib` 等本机路径来判断远程构建问题。
- 不要用本机性能数据推断远程 `riscv64` 性能。
- 不要覆盖或删除已有测试报告、构建日志、补丁和中间结论，除非用户明确要求。

## 远程执行约定

- 如需执行命令，应通过 SSH 登录远程主机：`ssh bianbu26@10.0.91.143`。
- 远程源码根目录：`/home/bianbu26/deb_eigen_try`。
- 可参考远程目录 `/home/bianbu26/eigen_rvv/eigen_custom_test` 中的编译选项、测试方法和脚本风格。

## 开发目标

目标是重新编译 `libceres-dev`，验证 `eigen-spacemit` 相比系统 `libeigen3-dev` 是否能提升 Ceres 相关计算性能。

工作应按以下阶段推进：

1. 基线测试
   - 先测试系统现有 `libceres-dev` + 系统 `libeigen3-dev` 的性能。
   - 编写一个可复用的 Ceres 性能测试程序，不依赖临时手工命令。
   - 保留完整基线测试报告，包括硬件/系统信息、包版本、编译命令、运行命令、原始输出和结论。

2. 构建改造
   - 获取或使用 `libceres-dev` 对应源码包。
   - 调整构建配置，使 Ceres 使用 `eigen-spacemit` 而不是系统 `libeigen3-dev`。
   - 编译过程参考 `/home/bianbu26/eigen_rvv/eigen_custom_test` 中已有选项。
   - 保留构建日志、补丁、配置差异和生成的 `.deb` 包信息。

3. 系统替换
   - 用新构建的 Ceres 包替换远程系统已有 `libceres-dev` 相关包。
   - 替换前记录原始包版本和可回滚方案。
   - 替换后确认头文件、库文件和链接关系确实来自新构建包。

4. 对比测试
   - 使用同一个可复用测试程序重新测试 `libceres-dev` + `eigen-spacemit` 的性能。
   - 测试条件应尽量与基线一致，包括编译选项、运行次数、CPU 频率/负载状态和输入规模。
   - 保留新测试报告，并与基线报告进行对比。

5. 结论整理
   - 输出最终报告，明确说明是否加速、加速幅度、测试置信度、可能误差来源和后续建议。
   - 报告必须包含可复现实验步骤，方便后续重新运行。

## 推荐目录结构

建议在仓库内维护以下内容：

- `benchmarks/`：可复用 Ceres 性能测试程序和运行脚本。
- `reports/`：基线、新包和最终对比报告。
- `logs/`：构建日志、测试原始输出和环境采集结果。
- `patches/`：对 Debian 源码包或构建配置的补丁。
- `packages/`：必要时保存生成的 `.deb` 包清单或校验信息；若包体积较大，只记录路径和校验值。

## 测试与报告要求

- 基准测试程序应能重复运行，并支持记录多次运行结果。
- 报告中至少记录：
  - 日期时间；
  - 远程主机名和架构；
  - 系统版本；
  - `libceres-dev`、Ceres 运行库、Eigen 相关包版本；
  - 编译器版本；
  - 编译选项；
  - 测试命令；
  - 原始输出；
  - 汇总指标和结论。
- 性能结论不能只看单次运行，至少保留多次运行数据。
- 对比时优先使用同一测试程序、同一构建方式、同一输入规模和同一统计口径。
- 除了术语，描述性语句均使用中文

## 操作安全

- 修改系统包前，先记录当前包状态和回滚方法。
- 使用 `apt`、`dpkg`、`sudo` 或会改变远程系统状态的命令前，要确保命令目标明确。
- 不要清理远程系统上的源码包、构建产物或报告，除非用户明确要求。
- 如发现远程环境、包名、路径或架构信息与本文件不一致，先向用户确认再继续。
